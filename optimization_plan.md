# 高并发日志系统丢弃率优化计划

> 基线数据（2026-07-10 压测，112 核 Linux，g++ -O2）：
>
> | 线程 | lockfree 吞吐/s | p99 入队延迟 | 丢弃率 |
> |---:|---:|---:|---:|
> | 8 | 87 万 | 1 µs | ~68% |
> | 32 | 71 万 | 1 µs | ~72% |
> | 112 | 20 万 | 512 ns | ~90% |
>
> 瓶颈定位：单写盘线程排水能力约 70~100 万条/s，生产侧峰值约 200 万条/s，
> 队列（8192 格）在压测开始约 6ms 后灌满，此后入队大量失败。
>
> **总原则：每次只改一个变量 → `make clean && make bench` 回归 → 记录数据 → 再改下一个。**

## 优化 1：提高单文件行数上限（性价比最高）

| 项目 | 内容 |
|---|---|
| 改动点 | `log_common.h:28` `MAX_LINES_PER_FILE`：`1000` → `100000` |
| 原理 | 每 1000 行触发一次 `flush+close+open`（`log_writer.cpp:38-42`），都是系统调用级重操作。基线压测切了 250+ 次文件，切分开销占满写盘线程时间片 |
| 步骤 | ① 改常量 → ② 重编译 → ③ `make bench` → ④ 确认 `bench_logs` 下文件数量明显减少 |
| 预期 | 写盘吞吐显著上升，丢弃率明显下降；文件数从数百个降到个位数 |
| 风险 | 单文件变大（10 万行 ≈ 数十 MB），查看/传输略笨重；可接受 |
| 回滚 | 改回常量即可 |
| 进阶 | 后续可改为按文件大小（如 64MB）切分，而不是按行数 |

## 优化 2：flush 改为按时间/按等级（几行代码）

| 项目 | 内容 |
|---|---|
| 改动点 | `logger.cpp` `thread_func()`：删掉"每批必 `writer_->flush_all()`"，改为定时刷 |
| 原理 | `BATCH_SIZE=32`，现在每 32 条就强刷一次，把 `ofstream` 用户态缓冲完全废掉 |
| 方案 | 记录上次 flush 时间戳，`steady_clock` 距上次 ≥100ms 才 `flush_all()`；循环退出前收尾必刷一次（现有收尾逻辑保留） |
| 可选增强 | 遇到 `LogLevel::ERROR` 的记录立即 flush（重要日志不留在缓冲里） |
| 预期 | 写盘吞吐上升；配合优化 1 效果叠加 |
| 风险 | 进程崩溃时最多丢最后 100ms 未刷缓冲的日志——这是吞吐 vs 可靠性的**主动取舍**，需在 README 注明 |
| 回滚 | 恢复每批 flush |
| 验证 | 压测中途 `kill -9`，检查文件尾部丢失量是否 ≤100ms 的写入量 |

## 优化 3：扩大队列容量（一个数字）

| 项目 | 内容 |
|---|---|
| 改动点 | `log_common.h:24` `LOG_QUEUE_CAPACITY`：`8192` → `65536`（**必须保持 2 的幂**，无锁队列取模依赖它） |
| 原理 | 容量估算公式：`所需容量 ≈ (进水速度 − 排水速度) × 突发时长`。基线：130 万/s × 0.3s ≈ 40 万格；65536 格可吸收约 50ms 突发 |
| 内存代价 | `sizeof(LogRecord)` ≈ 520B × 65536 ≈ 34MB，可接受；若做完优化 1/2 后仍不够，可再升到 262144（≈136MB） |
| 预期 | 突发型负载丢弃率大幅下降；对持续过载只能延迟丢弃开始时间，不能根治 |
| 风险 | 仅内存占用；注意两套队列（mutex/lockfree）都用该常量 |
| 回滚 | 改回常量 |

## 优化 4：加大批量出队（一个数字）

| 项目 | 内容 |
|---|---|
| 改动点 | `log_common.h:26` `BATCH_SIZE`：`32` → `512` |
| 原理 | 批次越小，写盘线程唤醒次数越多，条件变量/调度固定开销摊薄越差；512 时固定开销被摊薄 16 倍 |
| 注意 | `thread_func()` 中 `LogRecord batch[BATCH_SIZE]` 是栈数组，512×520B ≈ 266KB，仍在默认栈（8MB）安全范围内；若继续加大建议改成成员数组或堆分配 |
| 预期 | 高负载下写盘吞吐小幅上升（此项收益小于 1/2/3，属于锦上添花） |
| 回滚 | 改回常量 |

## 优化 5：消除每条日志的堆分配（中等工作量）

| 项目 | 内容 |
|---|---|
| 改动点 | `log_writer.cpp:47` `xor_encrypt_to_hex` 返回 `std::string` → 改为写入调用方提供的复用缓冲区（`log_crypto.h/.cpp` 接口同步调整） |
| 原理 | 现在每条日志一次 malloc + 一次拷贝，且 hex 编码让落盘字节数翻倍；换成栈/成员缓冲区后写盘热路径零分配 |
| 方案 | `void xor_encrypt_to_hex(const char* in, char* out, size_t out_cap);`，`LogWriter` 持有一个 `char buf[LOG_TEXT_CAPACITY*2+2]` 复用 |
| 预期 | 写盘吞吐上升 10%~30%（分配器压力消失） |
| 风险 | 接口改动牵连 `test_business.cpp` 中解密测试，需同步修改并跑 `make test` 确认 5 项测试仍全过 |
| 回滚 | git revert 该提交（建议此项单独提交） |

## 优化 6：分级背压——重要日志绝不丢（架构级）

| 项目 | 内容 |
|---|---|
| 改动点 | `logger.cpp` `vwrite()` 中 `try_push` 失败后的处理逻辑 |
| 原理 | 丢弃率不需要是 0，需要的是**重要日志丢弃率为 0**。现在是无差别 fail-fast |
| 方案 | 按等级分层：`ERROR` → 自旋+yield 直到入队成功（阻塞）；`WARN/INFO` → 重试 N 次（如 100 次 yield）后丢弃；`DEBUG` → 满即丢 |
| 预期 | ERROR 丢弃率 = 0；整体丢弃率下降；代价是过载时业务线程 p99 入队延迟从 µs 级升到 ms 级 |
| 风险 | 写盘线程若卡死（磁盘故障），ERROR 阻塞会传导到业务线程——需设置最长阻塞上限（如 10ms）兜底 |
| 验证 | 压测后按等级分别统计丢弃数（需给 `Metrics` 增加按等级计数） |

## 优化 7：多消费者分片（架构级，最后考虑）

| 项目 | 内容 |
|---|---|
| 改动点 | `Logger` 内部：单队列+单写盘线程 → 按 `LogCategory` 三个队列+三个写盘线程 |
| 原理 | 三类日志（application/operation/error）本就写三套独立文件（`log_writer.cpp:8-11`），互不共享文件句柄，天然可并行，无需加锁，排水能力理论 ×3 |
| 进阶方案 | 每个生产线程一个 SPSC 队列，写盘线程轮询汇总——同时消除生产者之间对队列头尾指针的缓存行争抢（这是 112 线程时吞吐暴跌到 20 万/s 的根因） |
| 预期 | 排水能力成倍提升；高线程数下生产侧吞吐恢复 |
| 风险 | 改动面大（Logger/LogWriter/Metrics 都要动），跨类别日志的全局时间顺序不再严格保证；必须最后做，且此前 1~5 已量化收益 |
| 前置条件 | 完成优化 1~4 并回归后，若丢弃率仍不达标才启动 |

## 实施顺序与验收标准

```
优化1 → 回归 → 优化3 → 回归 → 优化2 → 回归 → 优化4 → 回归
   （预期到此：本压测负载下丢弃率 ≈ 0）
→ 若持续过载场景仍丢：优化5 → 优化6 → 优化7
```

每轮回归必做三件事：

1. `make test` —— 5 项单元测试全过（正确性不回退）
2. `make bench` —— 记录 吞吐/p99/丢弃率 三个数，与上一轮对比
3. 完整性抽查 —— 入队数 == 落盘行数（可用扩展压测驱动校验）

| 验收项 | 目标 |
|---|---|
| 8~32 线程压测丢弃率 | < 1%（基线 68%~72%） |
| ERROR 日志丢弃率（做完优化 6 后） | = 0 |
| p99 入队延迟（lockfree） | 保持 µs 级不劣化 |
| 单元测试 | 5/5 通过 |
